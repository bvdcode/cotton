# 28. Glossary

This glossary defines the domain and technical vocabulary used throughout the Cotton Cloud engineering wiki. Every entry is grounded in the actual source code, names the exact identifier (class, enum, field, route, or constant) as it appears in the repository, and points to the relevant file(s) and/or other wiki sections. Where the `README.md` and the code disagree, the entry states what the code actually does. Terms are alphabetized; cross-references to other terms use *italic prose* and to other documentation sections use their plain section title.

## A–C

### AES-GCM
Cotton's symmetric authenticated-encryption primitive, implemented by `AesGcmStreamCipher` in `src/Cotton.Crypto/AesGcmStreamCipher.cs`. It is a streaming cipher with per-file key wrapping and per-chunk authentication: a random per-file content-encryption key is wrapped (AES-GCM) under the *MasterEncryptionKey* and stored in the file header, and the plaintext is split into chunks (default `DefaultChunkSize` = 1 MiB, bounded by `MinChunkSize` = 8 KiB and `MaxChunkSize` = 64 MiB) that are each encrypted with a fresh nonce and a 16-byte tag (`TagSize`). Constants: `KeySize` = 32, `NonceSize` = 12, `TagSize` = 16. The cipher is wired into the storage path through `CryptoProcessor` (`src/Cotton.Storage/Processors/CryptoProcessor.cs`). See the *Cryptography Engine* section.

### Bridge mode (Cotton Bridge)
An optional remote helper service hosted by the project (`Cotton Bridge`, base URL `https://bridge.cottoncloud.dev/api/v1/`, defined in `src/Cotton.Shared/Constants.cs`) used for features a self-hosted instance may not run locally. It backs the `Cloud` value of several mode enums: `EmailMode.Cloud` (relay email through the bridge, `src/Cotton.Database/Models/Enums/EmailMode.cs`), `ComputionMode.Cloud` (offload compute, `src/Cotton.Database/Models/Enums/ComputionMode.cs`), and `GeoIpLookupMode.CottonCloud` (geolocation, `src/Cotton.Database/Models/Enums/GeoIpLookupMode.cs`). Concrete consumers include `CottonPublicEmailProvider`, `GeoLookupService`, and `CollectPerformanceJob` (telemetry). Bridge connectivity health is probed by `SettingsProvider.CheckCottonBridgeHealthAsync`. Note: the enums also define a `Remote`/`Custom` value for self-supplied endpoints; the bridge is only the `Cloud` option.

### Canonical writer
`DatabaseIntegrityCanonicalWriter` (`src/Cotton.Server/Services/DatabaseIntegrity/DatabaseIntegrityCanonicalWriter.cs`) serializes an entity's security-sensitive fields into a deterministic, machine-independent byte payload before that payload is HMAC-signed. The format is deliberately rigid — it is prefixed with the magic `Cotton.DbIntegrity.Row.v1`, UTF-8 strings are length-prefixed, integers are big-endian, nullables carry an explicit presence byte, and dictionaries are key-sorted — so the same logical row always produces the same bytes regardless of machine, database provider, or JSON serializer version. It is the input side of the *Database integrity signature*; per-entity field selection is supplied by each *Integrity descriptor*.

### Chunk
The unit of content-addressed storage and the `Chunk` entity (`src/Cotton.Database/Models/Chunk.cs`, table `chunks`). Its primary key is `Hash`, the SHA-256 digest of the chunk's plaintext bytes; it records `PlainSizeBytes` (size before transforms), `StoredSizeBytes` (size after the *Storage pipeline* compresses and encrypts it), `CompressionAlgorithm`, and a nullable `GCScheduledAfter` timestamp (indexed) that drives *Garbage collection*. A chunk is referenced by ordered *FileManifestChunk* rows and guarded by *ChunkOwnership* rows. The maximum upload size is enforced by `CottonServerSettings.MaxChunkSizeBytes` and `AesGcmStreamCipher.MaxChunkSize` in `ChunkController` (`src/Cotton.Server/Controllers/ChunkController.cs`). See the *Content-Addressed Storage* section.

### ChunkOwnership
The `ChunkOwnership` entity (`src/Cotton.Database/Models/ChunkOwnership.cs`, table `chunk_ownerships`) links an owning user (`OwnerId`, from `BaseOwnedEntity`) to a `ChunkHash`, with a unique index on `(OwnerId, ChunkHash)`. It is a proof-of-ownership guard: a user may only reference (e.g. attach to a manifest, or learn that a chunk already exists) a chunk for which they hold an ownership row. `ChunkController.CheckChunkExists` returns whether a chunk is in storage only after confirming the caller owns it, and `ChunkIngestService.EnsureOwnershipAsync` (`src/Cotton.Server/Services/ChunkIngestService.cs`) inserts the row on every successful ingest. Ownership rows are deleted alongside the chunk during *Garbage collection*. See the *Content-Addressed Storage* section.

### ConcatenatedReadStream
`ConcatenatedReadStream` (`src/Cotton.Storage/Streams/ConcatenatedReadStream.cs`) is an internal `Stream` that reassembles a logical file on read by concatenating its ordered chunk streams pulled through the *Storage pipeline*. Given the ordered chunk hashes and an optional `PipelineContext`, it materializes a chunk index; it advertises `CanSeek` (and a real `Length`) only when the context supplies `ChunkLengths`, in which case it can binary-seek to the right chunk and offset for HTTP range requests, otherwise it is forward-only. This is the read-side counterpart to the *FileManifestChunk* ordering.

### Content addressing
The foundational storage model: a blob's storage key *is* a hash of its plaintext content rather than an arbitrary identifier. Cotton hashes chunk plaintext with SHA-256 (`Hasher`, `src/Cotton.Server/Services/Hasher.cs`; `ChunkIngestService` computes `SHA256.HashData`), stores the chunk under the lowercase-hex of that hash, and verifies uploads against the client-claimed hash with `CryptographicOperations.FixedTimeEquals`. Identical content therefore collapses to a single stored object (see *Manifest reuse / deduplication*) and any tamper changes the address. The sharded on-disk/S3 layout derived from the hash is produced by `StorageKeyHelper.GetSegments` (`src/Cotton.Storage/Helpers/StorageKeyHelper.cs`). See the *Content-Addressed Storage* and *System Overview & Design Philosophy* sections.

## D–G

### Database integrity signature
A per-row HMAC stored in two extra columns — `integrity_version` and `integrity_mac` (constants in `src/Cotton.Database/Integrity/DatabaseIntegrityColumns.cs`) — that lets Cotton detect tampering by an attacker who has only database access. `DatabaseIntegrityChangeSigner` (`src/Cotton.Server/Services/DatabaseIntegrity/DatabaseIntegrityChangeSigner.cs`) signs every `Added`/`Modified` protected entity in `SaveChanges`, and on update it first re-verifies the *original* row state, so a silent direct-SQL edit fails the next time Cotton touches the row. The MAC key is derived from the master key (`DatabaseIntegrityKeyProvider`); the signed bytes come from the *Canonical writer*; per-entity field choice comes from the *Integrity descriptor*. See the *Database Integrity* section.

### Download token
The `DownloadToken` entity (`src/Cotton.Database/Models/DownloadToken.cs`, table `download_tokens`) is a temporary, opaque, unique (`Token`) credential authorizing direct download of one file (`NodeFileId`). It carries a presented `FileName`, optional `ExpiresAt`, the creating user (`CreatedByUserId`), and a `DeleteAfterUse` flag for single-use links. The actual sweep (deletion) of expired tokens is performed by `DownloadTokenRetentionJob` (`src/Cotton.Server/Jobs/DownloadTokenRetentionJob.cs`), which removes tokens expired for more than 30 days; `DownloadTokenExpirationService` (`src/Cotton.Server/Services/DownloadTokenExpirationService.cs`) is the bulk-expire helper that sets `ExpiresAt` on a user's active tokens. It is the file-level analogue of the folder-level *Share token*. See the *Sharing & Tokens* section.

### Dust chunk
Informal term for a tiny stored chunk whose per-chunk encryption/compression header and metadata overhead are large relative to its payload. Cotton does not have a `DustChunk` type; the relevant levers are the AES-GCM chunk-size floor `AesGcmStreamCipher.MinChunkSize` (8 KiB) and the encryption chunk size chosen by `SettingsEncryptionChunkSizeProvider`, which keep stored chunks from being pathologically small. Contrast with *Orphan chunk* (a chunk with no live references).

### Encrypted preview hash
A preview whose storage hash is itself encrypted so the server cannot serve it without master-key-derived decryption. On `FileManifest` (`src/Cotton.Database/Models/FileManifest.cs`) the field `SmallFilePreviewHashEncrypted` holds the encrypted hash, while `SmallFilePreviewHash`/`LargeFilePreviewHash` hold plain hashes for previews that may be served directly. `FileManifest.GetPreviewHashEncryptedHex()` builds a row-scoped public token by concatenating the `PreviewTokenPrefix` (`'f'` for manifests; `'u'` for user avatars via `User.AvatarPreviewTokenPrefix`), the row id, and the encrypted hash hex; `PreviewController` (`src/Cotton.Server/Controllers/PreviewController.cs`) parses that token and decrypts the hash before reading from storage. See the *Previews & Thumbnails* section.

### File version
Cotton models historical versions without a separate version table: every version is an ordinary `NodeFile` row, and rows belonging to one lineage share `OriginalNodeFileId` (`src/Cotton.Database/Models/NodeFile.cs`). `FileVersionService` (`src/Cotton.Server/Services/FileVersionService.cs`) treats the row whose `Id == OriginalNodeFileId` as the immutable original; `FileVersionService.IsHistoricalVersion` returns true for any other row in the lineage. Because versions are plain `NodeFile` rows pointing at immutable *FileManifest* content, ownership, quota accounting, storage cleanup, and download authorization all reuse the normal file model. Retention is handled by `FileVersionRetentionService`. See the *File Versioning* section.

### FileManifest
The `FileManifest` entity (`src/Cotton.Database/Models/FileManifest.cs`, table `file_manifests`) represents one piece of immutable file content, independent of where it appears in any tree. It carries `ProposedContentHash` and `ComputedContentHash` (both uniquely indexed — see *ProposedContentHash vs ComputedContentHash*), `ContentType` (a `citext` MIME type), `SizeBytes`, preview hashes, and preview-generation bookkeeping (`PreviewGeneratorVersion`, `PreviewGenerationError`). Its ordered chunk list lives in *FileManifestChunk* rows; one manifest can back many visible *NodeFile* entries, which is what makes content reuse possible. See the *Content-Addressed Storage* section.

### FileManifestChunk
The ordered join entity between a *FileManifest* and its *Chunk*s (`src/Cotton.Database/Models/FileManifestChunk.cs`, table `file_manifest_chunks`). Each row holds `FileManifestId`, `ChunkHash`, and a zero-based `ChunkOrder`; a unique index on `(FileManifestId, ChunkOrder)` enforces that the ordering is well-defined and gap-free in intent, and `(ChunkHash)` is indexed for reverse lookups. Both the chunk and manifest navigations use `DeleteBehavior.Restrict`, so the database refuses to delete a chunk or manifest while a join row still references it — the structural backbone of the *Storage Lifetime Contract*. Reads walk these rows in `ChunkOrder` to feed a *ConcatenatedReadStream*. See the *Content-Addressed Storage* section.

### Garbage collection
The deferred deletion of unreferenced chunks and orphaned manifests, run by `GarbageCollectorJob` (`src/Cotton.Server/Jobs/GarbageCollectorJob.cs`), a Quartz job triggered every 6 hours (`[JobTrigger(hours: 6)]`, `[DisallowConcurrentExecution]`). One pass (`RunOnceAsync`) deletes orphaned manifests, clears the GC schedule for any chunk that became referenced or *protected* again, schedules newly-orphaned chunks by stamping `GCScheduledAfter` (default delay `ChunkGcDelayDays` = 7 days; shortened to 1 day under `StorageSpaceMode.Limited`, lengthened to 28 days under `Unlimited`), and finally deletes chunks whose schedule has elapsed. Deletion re-checks references at every step, reserves hashes in a static `CurrentlyDeletingChunks` set to coordinate with *ChunkIngestService* (which waits via `GarbageCollectorJob.IsChunkBeingDeleted`), deletes DB rows in a transaction, then deletes from storage with up to `StorageDeleteConcurrency` = 8 parallel calls. The eligibility rules come from *ChunkUsageService*. See the *Garbage Collection* section.

## H–N

### HLS / rendition / transcode
Cotton's adaptive video playback. `HlsRendition` (`src/Cotton.Server/Services/HlsRenditionProfile.cs`) is an enum of quality levels — `Source`, `High`, `Medium`, `Low` — and `HlsRenditionProfile` describes the per-rendition encoder plan. `VideoTranscoder` (`src/Cotton.Server/Services/VideoTranscoder.cs`) transcodes one HLS *segment* at a time with ffmpeg, `HlsManifestBuilder` produces the `.m3u8` playlists, `HlsSegmentCache` caches produced segments, and `VideoPlaybackResolver` chooses a rendition. Because ffmpeg needs URL-based seeking over the decrypted source, the transcoder serves the source through a loopback *RangeStreamServer*. See the *Media Transcoding & HLS* section.

### Integrity descriptor
An implementation of `IDatabaseIntegrityDescriptor` (`src/Cotton.Server/Services/DatabaseIntegrity/IDatabaseIntegrityDescriptor.cs`) that defines, for one EF entity type, which fields are security-sensitive and how to turn the row into a canonical signed payload. It exposes `EntityType`, a stable `EntityName`, a `SchemaVersion`, `GetEntityKey`, and `WriteCanonicalData` (which drives the *Canonical writer*). Concrete descriptors live in `src/Cotton.Server/Services/DatabaseIntegrity/Descriptors/` — e.g. `ChunkIntegrityDescriptor`, `FileManifestIntegrityDescriptor`, `FileManifestChunkIntegrityDescriptor`, `NodeIntegrityDescriptor`, `NodeFileIntegrityDescriptor`, `UserIntegrityDescriptor`, `DownloadTokenIntegrityDescriptor`, `NodeShareTokenIntegrityDescriptor`, `OidcProviderIntegrityDescriptor`, and more — and are looked up at runtime through `IDatabaseIntegrityDescriptorRegistry`. Adding a field here is what makes a database-only attacker unable to edit that field silently. See the *Database Integrity* section.

### JWT access token
A short-lived signed bearer token issued by `AuthSessionIssuer.CreateAccessToken` (`src/Cotton.Server/Services/AuthSessionIssuer.cs`) after an authentication factor succeeds. It carries the user identity and a session id claim (`JwtRegisteredClaimNames.Sid` / `ClaimTypes.Sid`) tying it to a *Session*; the signing key is the randomly generated `JwtSettings:Key` produced at startup in `ConfigurationBuilderExtensions.AddCottonOptions`. Revocation of still-valid access tokens is tracked by `SessionAccessTokenRevocationStore`/`SessionAccessTokenRevocationCache`. It is paired with, but distinct from, the *Refresh token*. See the *Authentication & Sessions* section.

### Layout
The `Layout` entity (`src/Cotton.Database/Models/Layout.cs`, table `layouts`) is one user-owned file tree (`BaseOwnedEntity` gives it `OwnerId`) with an `IsActive` flag and a collection of *Node*s. Layouts implement the git-like "layout vs content" split central to Cotton: a layout is the mutable, per-user namespace/structure, while the bytes live in immutable content-addressed *FileManifest*s and *Chunk*s. Navigation and traversal live in `Cotton.Topology` (`StorageLayoutService`, `LayoutNavigator`). See the *Layouts & Node Tree* and *System Overview & Design Philosophy* sections.

### Live reference
The condition that keeps a chunk from being garbage-collected: a chunk is *referenced* (live) when some database row points at it. `ChunkUsageService.WhereReferencedByDatabase` (`src/Cotton.Server/Services/ChunkUsageService.cs`) defines this precisely — a chunk is live if any `FileManifestChunk` references it, or any `FileManifest` uses it as `SmallFilePreviewHash`/`LargeFilePreviewHash`, or any `User` uses it as `AvatarHash`. `WhereUnreferencedByDatabase` is the negation. A live reference is one half of the *Storage Lifetime Contract*; the other half is being *protected* (e.g. backup manifest or the *Sentinel*). See the *Garbage Collection* section.

### Manifest reuse / deduplication
The reuse of a single stored *Chunk* or *FileManifest* across many file entries or users. At the chunk layer, `FileStoragePipeline.WriteAsync` (`src/Cotton.Storage/Pipelines/FileStoragePipeline.cs`) skips the whole processor pipeline when the content-addressed UID already exists, and `ChunkIngestService.TryReuseDeduplicatedChunkAsync` reuses an existing chunk row (subject to `CottonServerSettings.AllowCrossUserDeduplication`), only adding a new *ChunkOwnership* row. At the manifest layer, the unique indexes on `FileManifest.ProposedContentHash` and `ComputedContentHash` mean identical content resolves to one manifest that many *NodeFile* rows share. Cross-user dedup is gated by the `AllowCrossUserDeduplication` setting because it has privacy implications (existence of a chunk is observable). See the *Content-Addressed Storage* section.

### Mapster
The object-mapping library used to project EF entities to DTOs. Configuration is registered once via `MapsterConfig.Register()` in `src/Cotton.Server/Program.cs`, and handlers map with the `Adapt<T>()` extension (e.g. `user.Adapt<UserDto>()` in `src/Cotton.Server/Handlers/Users/AdminCreateUserRequest.cs`). Mapster sits at the boundary between the persistence model and the API contract.

### Master key
The single 32-character root secret from which all of Cotton's process-local cryptographic material is derived; constant `DefaultKeyLength` = 32 in `src/Cotton.Autoconfig/Extensions/ConfigurationBuilderExtensions.cs`. It is supplied either via the `COTTON_MASTER_KEY` environment variable (`MasterKeyEnvironmentVariable`) or interactively through the `/unlock` flow. `DeriveEncryptionSettings` HKDF-derives the *Pepper* (info `"CottonPepper"`) and *MasterEncryptionKey* (info `"CottonMasterEncryptionKey"`) from it; the raw key is then scrubbed from the process and user environment by `ClearMasterKeyEnvironmentVariable`. Changing its length or value after first use is irreversible and makes all derived-key data — including password hashes — unrecoverable. See the *Master Key, Autoconfig & Unlock Bootstrap* section.

### MasterEncryptionKeyId
The integer identifier of the active master-key derivation scheme, field `MasterEncryptionKeyId` on `CottonEncryptionSettings` (`src/Cotton.Shared/CottonEncryptionSettings.cs`), seeded to `DefaultMasterKeyId` = 1 by `ConfigurationBuilderExtensions`. It is embedded as `keyId` in the AES-GCM file/chunk headers and in the associated data (`AesGcmStreamFormat.InitAadPrefix`/`BuildKeyAad`, `src/Cotton.Crypto/Internals/AesGcmStreamFormat.cs`), so ciphertext is cryptographically bound to the key generation that produced it and a future key rotation can be distinguished from old data. See the *Master Key, Autoconfig & Unlock Bootstrap* and *Cryptography Engine* sections.

### Mediator handler (command/query)
Cotton uses the EasyExtensions.Mediator request/handler pattern for most application logic. A request is a small record implementing `IRequest<TResponse>` (or `IRequest`) and its handler implements `IRequestHandler<TRequest, TResponse>`; examples include `GetChildrenQuery` (`src/Cotton.Server/Handlers/Nodes/GetChildrenQuery.cs`), `DeleteNodeQuery`, `AdminCreateUserRequest`, and the WebDAV handlers under `src/Cotton.Server/Handlers/WebDav/`. Controllers and other handlers dispatch through `IMediator`/`ISender`. Each handler is one file co-locating the request shape with its behavior. See the *Request Pipeline & Handlers* section.

### Nonce (file prefix + chunk counter)
The 12-byte AES-GCM IV used per encrypted chunk, composed by `AesGcmStreamFormat.ComposeNonce` (`src/Cotton.Crypto/Internals/AesGcmStreamFormat.cs`) as `[4-byte per-file prefix][8-byte little-endian chunk counter]`. The random per-file prefix plus a monotonically increasing chunk counter guarantees nonces are unique within a file under one content-encryption key; the counter space is 64-bit and the code explicitly throws `InvalidOperationException` if it would reach `ulong.MaxValue` to prevent catastrophic IV reuse. The chunk index and plaintext length are also folded into the per-chunk AAD (`FillAadMutable`). See the *Cryptography Engine* section.

### Node
The `Node` entity (`src/Cotton.Database/Models/Node.cs`, table `nodes`) is a folder-like element inside one *Layout* tree. It has `LayoutId`, a nullable `ParentId` (null = layout root), a `NodeType` discriminator, a validated `Name`/`NameKey` pair, and an extensible `Metadata` dictionary. A unique index on `(LayoutId, ParentId, Type, NameKey)` prevents duplicate sibling names within a type. `SetParent` enforces that a child's parent has the same `OwnerId`, the same `LayoutId`, and a matching `NodeType`, which is why a `Default` subtree can never be re-parented under the `Trash` subtree and vice versa. See the *Layouts & Node Tree* section.

### NodeFile
The `NodeFile` entity (`src/Cotton.Database/Models/NodeFile.cs`, table `node_files`) is a named, visible file entry that lives inside a *Node* and points at immutable content via `FileManifestId`. It has its own validated `Name`/`NameKey`, `Metadata`, `OwnerId`, and `OriginalNodeFileId` (the lineage anchor used for *File version*s). Many `NodeFile` rows can reference the same *FileManifest* (the same bytes appearing in multiple places or versions). It also owns its *Download token* rows. See the *Layouts & Node Tree* and *File Versioning* sections.

### NodeType
The enum `NodeType` (`src/Cotton.Database/Models/Enums/NodeType.cs`) that separates independent node trees so they cannot be mixed as parent and child. Values: `Default = 0` (the normal user-visible file tree) and `Trash = 1` (the *Trash / soft delete* tree). It participates in the `Node` uniqueness index and in `Node.SetParent`'s parent-type check. See the *Trash & Soft Delete* section.

### NameKey
A normalized, case-insensitive lookup key derived from a display `Name`, stored as a PostgreSQL `citext` column on both `Node` and `NodeFile`. `SetName` validates and normalizes the human name via `NameValidator.TryNormalizeAndValidate` and derives the key via `NameValidator.GetNameKey` (`Cotton.Validators`); the resulting `NameKey` is what uniqueness indexes and lookups use, so "Report.PDF" and "report.pdf" collide as the same sibling name. Both `Name` and `NameKey` have private setters precisely to force callers through `SetName`. See the *Layouts & Node Tree* section.

### Notification
The `Notification` entity (`src/Cotton.Database/Models/Notification.cs`, table `notifications`) is one message delivered to a user: `Title`, optional `Content`, `UserId`, a `NotificationPriority`, optional `Metadata`, and a `ReadAt` timestamp. Notifications are created by services such as `StoragePressureGuard` and `CottonNotifications`/`EventNotificationService` and pushed to clients in realtime via *SignalR / EventHub* (`EventHub.NotificationMethod = "OnNotificationReceived"`). `NotificationPriority` (`src/Cotton.Database/Models/Enums/NotificationPriority.cs`) has values `None = 0`, `Low = 1`, `Medium = 2`, `High = 3`. See the *Notifications & Realtime Events* section.

## O–R

### OIDC / external identity
Cotton supports OpenID Connect single sign-on. `OidcProvider` (`src/Cotton.Database/Models/OidcProvider.cs`, table `oidc_providers`) is an admin-configured identity provider (issuer, `ClientId`, encrypted `ClientSecretEncrypted`, scopes, account-creation policy, profile/avatar sync flags). `UserExternalIdentity` (`src/Cotton.Database/Models/UserExternalIdentity.cs`, table `user_external_identities`) links one Cotton `User` to one provider `Subject` (the only stable external key), with unique indexes on `(ProviderId, Subject)` and `(UserId, ProviderId)`. The login dance is coordinated by `OidcAuthenticationService`, `OidcDiscoveryService`, `OidcProviderService`, and `OidcAvatarImportService`, with transient state in `OidcLoginState`. See the *External Identity (OIDC)* section.

### Orphan chunk
A *Chunk* with no *Live reference* and not otherwise protected — i.e. one matched by `ChunkUsageService.WhereUnreferencedByDatabase`. `GarbageCollectorJob.ScheduleOrphanedChunksAsync` stamps such chunks with a future `GCScheduledAfter`, and `DeleteScheduledChunksAsync` removes them once that deadline passes and a re-check confirms they are still orphaned. An *orphaned manifest* is the analogous concept for `FileManifest` rows with no `NodeFile` (`DeleteOrphanedManifestsAsync`). See the *Garbage Collection* section.

### Paranoia Mode
An operator-facing hardening posture documented in `README.md` (the "Paranoia Mode" section) rather than a single code switch. In code it corresponds to running the official Docker image as the non-root `app` user, disabling .NET diagnostics, and requesting `PR_SET_DUMPABLE=0`; the relevant environment variable read by Cotton is `COTTON_PROCESS_HARDENING`, applied by `LinuxProcessHardening`/`LinuxContainerSecurity` (`src/Cotton.Server/Services/`). The remaining "paranoia" knobs (`cap_drop: ALL`, `no-new-privileges`, read-only root, seccomp/AppArmor) are Docker/Compose-level settings outside Cotton. The visible state of these signals is reported by `SecurityDiagnosticsService` at `GET /api/v1/server/security/status`. See the *Security Hardening* section.

### Passkey / WebAuthn
Cotton supports phishing-resistant WebAuthn passkeys. `UserPasskeyCredential` (`src/Cotton.Database/Models/UserPasskeyCredential.cs`, table `user_passkey_credentials`) stores one registered credential: `CredentialId` (unique), the COSE `PublicKey`, `UserHandle`, a `SignatureCounter` for cloned-credential detection, `Transports`, `AaGuid`, backup-eligibility/backed-up flags, and `AttestationFormat`. Registration and assertion verification live in `PasskeyService` (`src/Cotton.Server/Services/PasskeyService.cs`). See the *Authentication & Sessions* section.

### Pepper
A 32-byte secret derived from the *Master key* with HKDF info `"CottonPepper"` (`ConfigurationBuilderExtensions.DeriveEncryptionSettings`), surfaced as `CottonEncryptionSettings.Pepper` (`src/Cotton.Shared/CottonEncryptionSettings.cs`). It is an additional server-held input to password/secret hashing — distinct from a per-record salt — so that a stolen database alone cannot mount offline attacks without also stealing the master key. Like all derived material it depends on the unchangeable 32-char key length. See the *Master Key, Autoconfig & Unlock Bootstrap* section.

### Pipelines (System.IO.Pipelines)
The high-throughput buffer-management API Cotton uses on hot streaming paths to avoid intermediate allocations and to backpressure producers against consumers. `CompressionProcessor.WriteAsync` (`src/Cotton.Storage/Processors/CompressionProcessor.cs`) builds a `System.IO.Pipelines.Pipe` with explicit `pauseWriterThreshold`/`resumeWriterThreshold` and exposes the reader as a stream while a background task feeds the Zstd compressor; the crypto pipelines in `src/Cotton.Crypto/Internals/Pipelines/` (`EncryptionPipeline`, `DecryptionPipeline`, `ReorderBuffer`) use the same primitives to parallelize per-chunk AES-GCM while preserving order. See the *Cryptography Engine* section.

### Preview
A generated thumbnail/preview image for a *FileManifest*. Manifests carry small and large preview hashes plus generation bookkeeping (`PreviewGeneratorVersion`, `PreviewGenerationError`); previews are produced by `GeneratePreviewJob` (`src/Cotton.Server/Jobs/GeneratePreviewJob.cs`) using the `Cotton.Previews` library and served by `PreviewController`. Previews may be stored either as plain hashes (servable directly) or as an *Encrypted preview hash* (decrypted on demand). See the *Previews & Thumbnails* section.

### ProposedContentHash vs ComputedContentHash
Two uniquely-indexed hash columns on `FileManifest` (`src/Cotton.Database/Models/FileManifest.cs`). `ProposedContentHash` is the client-asserted content hash used for early deduplication and lookup *before* the server has all the bytes; `ComputedContentHash` (nullable) is the hash the server itself computes and trusts for verification once the content is assembled. Keeping both — and both unique — lets Cotton dedup optimistically on the client claim while still detecting a mismatch when the server-side value is filled in. This mirrors the chunk-level verification in `ChunkIngestService.UpsertChunkAsync`, which compares a `computedHash` against the caller's `expectedHash` with `FixedTimeEquals`. See the *Content-Addressed Storage* section.

### Quartz job
A scheduled background task implementing `Quartz.IJob`, decorated with `EasyExtensions.Quartz.Attributes.JobTrigger` for its cadence. Examples: `GarbageCollectorJob` (`[JobTrigger(hours: 6)]`, `src/Cotton.Server/Jobs/GarbageCollectorJob.cs`), `GeneratePreviewJob`, `CollectPerformanceJob`, and `BackfillChunkStoredSizeJob`. `[DisallowConcurrentExecution]` prevents overlapping runs of the same job. See the *Scheduled Jobs* section.

### Quota
A logical per-user storage limit on *referenced file data*, enforced by `UserStorageQuotaService` (`src/Cotton.Server/Services/UserStorageQuotaService.cs`). Usage is the sum of `FileManifest.SizeBytes` over a user's `NodeFile` rows (`GetUsedBytesAsync`), cached in-process and in `IMemoryCache` for 15 minutes; the limit default is `CottonServerSettings.DefaultUserStorageQuotaBytes`. Quota is deliberately billed against *logical file references*, not raw chunk uploads — raw chunk admission is governed instead by the *Storage pressure guard*, because chunking is an internal storage detail and a chunk only becomes user-owned data once a file reference reaches it. See the *Quotas & Storage Pressure* section.

### RangeStreamServer
`RangeStreamServer` (`src/Cotton.Previews/Http/RangeStreamServer.cs`) is a short-lived loopback `HttpListener` that serves a single seekable stream with HTTP `Range` support and a random URL token, used so external tools that require URL-based seeking (notably ffmpeg during *HLS / rendition / transcode* and preview generation) can read a decrypted source without it being written to disk. It requires the wrapped stream to be seekable. See the *Media Transcoding & HLS* section.

### Refresh token rotation
Cotton issues long-lived refresh credentials as `ExtendedRefreshToken` rows and rotates them: `AuthSessionIssuer.CreateRefreshTokenAsync` (`src/Cotton.Server/Services/AuthSessionIssuer.cs`) generates a fresh random token (length `AuthController.RefreshTokenLength`), stores only its hash (`HashRefreshToken`) with a `SessionId`, and sets it as an HTTP cookie. On use the token is exchanged for a new *JWT access token* and a new refresh token, and revoked refresh tokens are managed by `RefreshTokenRevocationService`. A refresh token is bound to a *Session*; revoking the session invalidates it. See the *Authentication & Sessions* section.

## S–Z

### Segment (S3)
On the S3 backend, Cotton stores each chunk as a single bucket object whose key is the *sharded* form of the chunk UID: `GetS3Key` (`src/Cotton.Storage/Backends/S3StorageBackend.cs`) calls `StorageKeyHelper.GetSegments` to split the hex UID into two 2-character shard prefixes and a remainder, producing `p1/p2/filename.ctn`. The "segments" are these key components; `ListAllKeysAsync` reverses the split to reconstruct UIDs from `.ctn` objects. (Note: despite the prose name, the current S3 backend buffers each chunk to a temp file and does one `PutObjectRequest` — it does not use S3 multipart upload.) Not to be confused with an HLS *segment* (a transcoded video fragment). See the *Storage Backends* section.

### Sentinel
A small encrypted marker object that proves a submitted *Master key* matches this instance, implemented by `MasterKeySentinelStore` (`src/Cotton.Server/Services/MasterKeySentinelStore.cs`). Its storage key is the hash of the logical key `"cotton.master-key.sentinel.v1"` (`SentinelStorageKey`). At startup `ValidateOrInitializeAsync` either decrypts and validates the existing sentinel, creates one for a fresh instance, or — when an existing sentinel cannot be decrypted but the *compatibility probe* finds the key matches real encrypted data — repairs it. The sentinel storage key is one of the *protected storage keys* the garbage collector must never delete. See the *Master Key, Autoconfig & Unlock Bootstrap* section.

### Session
The server-side notion of one signed-in client, identified by a `SessionId` carried in the `Sid` JWT claim. The *JWT access token* and the *Refresh token* both reference the same session; `EventHub` groups realtime connections by `auth-session:{userId:N}:{sessionId}` (`EventHub.GetSessionGroupName`) so a session can be force-disconnected, and `SessionRevocationNotifier`/`SessionAccessTokenRevocationStore` revoke a session's still-valid access tokens (signalling clients via `EventHub.SessionRevokedMethod = "SessionRevoked"`). See the *Authentication & Sessions* section.

### Share token (NodeShareToken)
The `NodeShareToken` entity (`src/Cotton.Database/Models/NodeShareToken.cs`, table `node_share_tokens`) is the folder-level sharing credential, explicitly parallel to the file-level *Download token*. It stores the shared folder's `Name` at share time, a unique opaque `Token` used in the share URL, the shared `NodeId`, an optional `ExpiresAt` (null = never), and the creating user. Its `Node` navigation uses `DeleteBehavior.Cascade`, so deleting the shared folder removes the share. See the *Sharing & Tokens* section.

### Storage backend
The lowest storage layer, `IStorageBackend` (`src/Cotton.Storage/Abstractions/IStorageBackend.cs`): an opaque object store keyed by normalized content UID that knows nothing about users, files, manifests, keys, or quotas. Implementations are `FileSystemStorageBackend` (local disk, `src/Cotton.Storage/Backends/FileSystemStorageBackend.cs`) and `S3StorageBackend` (`src/Cotton.Storage/Backends/S3StorageBackend.cs`); selection corresponds to the `StorageType` enum (`Local = 0`, `S3 = 1`, `src/Cotton.Database/Models/Enums/StorageType.cs`). Backends are obtained through `IStorageBackendProvider`; capacity-aware backends also implement `IStorageCapacityReporter` (consumed by the *Storage pressure guard*). See the *Storage Backends* section.

### Storage Lifetime Contract
The database-driven invariant that a *Chunk* is never deleted while anything still depends on it, and is deleted promptly once nothing does. It is enforced at three layers: the EF `DeleteBehavior.Restrict` on `FileManifestChunk` → `Chunk` and `FileManifest` (the database refuses orphaning deletes), the reachability rules in `ChunkUsageService` (*Live reference* and *protected storage keys*), and the deferred, re-checked deletion in `GarbageCollectorJob` (a 7-day default grace window, re-verification before deletion, and the `CurrentlyDeletingChunks` reservation that blocks concurrent re-ingest). See the *Content-Addressed Storage* and *Garbage Collection* sections.

### Storage pipeline
`IStoragePipeline` (`src/Cotton.Storage/Abstractions/IStoragePipeline.cs`), implemented by `FileStoragePipeline` (`src/Cotton.Storage/Pipelines/FileStoragePipeline.cs`), composes the ordered *Storage processor*s around the selected *Storage backend*. On write, processors run in descending `Priority` (compression then encryption … toward the backend) and the pipeline short-circuits when the content-addressed UID already exists (deduplication); on read they run in ascending `Priority` to undo the transforms. Writes are bounded by a static semaphore of `Environment.ProcessorCount`. A `PipelineContext` carries optional per-operation metadata (e.g. `ChunkLengths`, `FileSizeBytes`) used by *ConcatenatedReadStream*. See the *Storage Pipeline & Processors* section.

### Storage pressure guard (HTTP 507)
`StoragePressureGuard` (`src/Cotton.Server/Services/StoragePressureGuard.cs`) protects the backend from writes that would push free space below a configured reserve. It reads a (briefly cached) capacity snapshot via `IStorageCapacityReporter`, immediately subtracts in-flight reservations so a burst of chunks cannot all pass against the same stale snapshot, and on insufficient space throws `StoragePressureException` and notifies admins once per cooldown. `ChunkController` translates that exception into HTTP `507 Insufficient Storage` ("Uploads are temporarily paused"). It is configured by `StoragePressureOptions` (`src/Cotton.Server/Models/Configuration/StoragePressureOptions.cs`) and is the chunk-upload counterpart to the user-level *Quota*. See the *Quotas & Storage Pressure* section.

### Storage processor
`IStorageProcessor` (`src/Cotton.Storage/Abstractions/IStorageProcessor.cs`): a transform applied to a blob stream as it moves between Cotton and the backend, ordered by an integer `Priority` (lower = closer to the backend). The two built-in processors are `CompressionProcessor` (Zstandard, `Priority = 10000`, `src/Cotton.Storage/Processors/CompressionProcessor.cs`) and `CryptoProcessor` (AES-GCM, `Priority = 1000`, `src/Cotton.Storage/Processors/CryptoProcessor.cs`). Because compression has the higher priority, on write the bytes are compressed first and then encrypted (so ciphertext, which does not compress, is never fed to the compressor). See the *Storage Pipeline & Processors* section.

### Trash / soft delete
Deleting a file or folder moves it into the user's `Trash` *Layout*/subtree (`NodeType.Trash`) rather than destroying data, so it can be restored. Restoration and trash bookkeeping are handled by `TrashRestoreCoordinator` and `TrashMetadataKeys` (`src/Cotton.Server/Services/`); the `NodeType` parent-matching rule (`Node.SetParent`) keeps the trash subtree structurally separate from the `Default` tree. Only after a chunk's last referencing `NodeFile`/manifest is truly gone does it become an *Orphan chunk* eligible for *Garbage collection*. See the *Trash & Soft Delete* section.

### Unlock / bootstrap token
The interactive startup path for supplying the *Master key* without an environment variable. `MasterKeyUnlockServer` (`src/Cotton.Server/Services/MasterKeyUnlockServer.cs`) runs a minimal pre-unlock web app that gates all non-unlock API requests until a key is submitted at `/api/v1/unlock`. To stop an attacker who can reach a freshly-started instance from supplying their own key, the first unlock within an initial window (`FirstUnlockWindow`) requires a **bootstrap token** (`GenerateBootstrapToken`) that is printed to the server console/logs; the `/unlock/status` endpoint reports `RequiresBootstrapToken` and the expiry. Submitted keys are run through the *Sentinel* and compatibility probe before the instance finishes starting. See the *Master Key, Autoconfig & Unlock Bootstrap* section.

### WebDAV
A standards-based remote file-access protocol Cotton exposes so OS file managers can mount a user's tree. Verb handlers live in `src/Cotton.Server/Handlers/WebDav/` (`WebDavPropFindQuery`, `WebDavGetFileQuery`, `WebDavPutFileRequest`, `WebDavMkColRequest`, `WebDavCopyRequest`, `WebDavMoveRequest`, `WebDavDeleteRequest`, `WebDavHeadQuery`), with path resolution, XML responses, and auth caching in `src/Cotton.Server/Services/WebDav/` (`WebDavPathResolver`, `WebDavXmlBuilder`, `WebDavAuthCache`). WebDAV authenticates with a per-user token whose hash is stored as `User.WebDavTokenPhc`. See the *WebDAV Access* section.

### Zstd compression
The Zstandard compression applied to stored blobs before encryption, via `CompressionProcessor` using `ZstdSharp` (`src/Cotton.Storage/Processors/CompressionProcessor.cs`). The default level is `DefaultCompressionLevel` = 1, bounded by the bundled build's `MinCompressionLevel`/`MaxCompressionLevel`; the active level at runtime comes from `SettingsCompressionLevelProvider`. Each `Chunk` records the algorithm it was stored with in `Chunk.CompressionAlgorithm` (`CompressionAlgorithm.Zstd`, from `EasyExtensions.Models.Enums`), allowing decompression to stay correct even if the default changes. See the *Storage Pipeline & Processors* section.
